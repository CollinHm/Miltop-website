# Miltop Challenge

## Doel
* Het product moet van goede kwaliteit zijn,
* aantrekkelijk zijn voor de doelgroep,
* volledig aan de eisen van de klant voldoen
* en voldoen aan de technische eisen.

## Technische eisen
* Pixel perfect (dus exact volgens het design)
* Animaties met GSAP: 100% (geen fouten, geen waarschuwingen)
* Copyright vrije assets (of zelfgemaakt) of met de juiste licentie en bronvermelding
* Geluid/muziek (let op rechten)
* Connectie met API voor gegevens over water
* Werkende game verwerkt op de website 
* Vue 3
* REST-backend om nieuwsitems en proefjes toe te voegen en op te halen (Node.js + Express)
* Online te bereiken (dus gehost op een server, niet alleen lokaal)
* Responsiveness (werkt goed op verschillende schermformaten)
* Getest met Lighthouse (score van minimaal 90 op performance, accessibility, best practices en SEO)
* UAT (User Acceptance Test) uitgevoerd en gedocumenteerd
* Productieklare code (leesbaar, goed gestructureerd, met comments)
* Duidelijke commit history met beschrijvende commit messages
* README.md met uitleg over het project, de installatie en de gebruikte technologieën
* FDD en TDD die zijn doorgevoerd in het project
* Duidelijke video van de werkende applicatie (max 25MB, screencast met voice-over)
* Duidelijke presentatie van het eindproduct

## Over Miltop
Miltop is een bedrijf dat zich bezighoudt met waterzuivering.
Ze willen graag het publiek bewust maken over klimaatadaptatie en het zuiveren van water.
Daarom hebben studenten van Mediavormgeving vorig jaar de opdracht gekregen om een concept te bedenken.
Een onderdeel was het beschilderen van een container (voor waterzuivering) op locatie (natuurtuin 't Bundertje in Veghel).
Maar ook een design voor een website die gedeeltelijk ook is uitgevoerd.

Deze opdrachtgever wil graag dat de website verder wordt uitgewerkt waaronder een game.

## Prepare
* Create a scrum board in `GitHub Projects` with these columns:
  sprint backlog (ToDo), in progress, testing, done
* Place the refined requirements in `GitHub Issues` (which will act as your backlog).
* Prioritize the backlog items. Use a label in `GitHub Issues` to indicate the priority.
* Have a Sprint Planning meeting and decide which issues in the backlog you will address in the coming sprint.
* Refine them!
* Move them to the sprint backlog column (ToDo) on the scrum board.

## Realise
Start a stand-up meeting every time you begin working on the project (which is probably each lesson):
* Open the scrum board
* Each team member answers these questions:
  * What did you do last time?
  * What will you do today?
  * Are there any impediments in your way?
* Update the scrum board.
* Start developing on the sprint backlog items (ToDo).

> ### Don't forget
> * Commit and push your changes regularly.
> * Make sure that your code is readable and has comments.
> * Make sure that your code follows the code conventions.

## Test
* Test every item during development.
* Write tests (test plan) based on the requirements.
* Let non-team members run these tests.
* Write down the results in the TEST_RESULT.md document.

## Review
* Show the result of each item in the `Done` column.
* Write down feedback you get in `REVIEW.md`.
* Add/change items in the backlog (in Issues) based on the feedback in `REVIEW.md`.

## Retrospective
* Have a retrospective meeting at the end of the sprint.
* Use the template file `RETROSPECTIVE.md` to write down the results of the retrospectives.

## Code Review
* You will review your code with your team and teacher.
* Write down the results in `CODE_REVIEW.md` in the `sintlucas` folder.

## Presentation & Assessment

* You will be assessed on the criteria in `sintlucas/ASSESSMENT.MD`.
